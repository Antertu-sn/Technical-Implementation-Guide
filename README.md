Настоящее руководство (Technical Implementation Guide) разработано в качестве прикладного дополнения к основному аналитическому докладу *«Архитектура системной памяти и защиты контекста в экосистеме IMARCH»*.

## Раздел 1. Дихотомия памяти: Контекст vs Системная память
Для корректной реализации архитектуры разработчикам необходимо строго разграничивать два технологических слоя обработки данных большой языковой моделью (LLM).
### 1.1. Контекст диалога (Краткосрочный слой)
*   **Жизненный цикл:** Ограничен временем существования одной сессии (одной конкретной ветки чата).
*   **Назначение:** Динамическая фиксация текущих соглашений, деталей текущего обсуждения и пользовательского инпута.
*   **Уязвимость перед лимитом токенов (Token Limit Exhaustion):** Любые инструкции, переданные в открытом виде с фронтенда как часть пользовательского диалога, неизбежно вытесняются и стираются по мере заполнения буфера памяти (скользящего окна контекста). При достижении лимита модель выполняет сброс (перезапись) контекста и возвращается к своим базовым «диким» настройкам, запуская галлюцинации и игнорируя ранее заданные правила.
### 1.2. Системная память (Долговременный слой на бэкенде)
*   **Жизненный цикл:** Бессрочный (Persistent).
*   **Назначение:** Архивное хранение верифицированных фактов, пользовательских предпочтений, устойчивого стиля общения и фундаментальных системных ограничений.
*   **Механизм работы:** Переносится между отдельными сессиями (ветками чатов) и принудительно инжектируется на уровне бэкенда при инициализации любого нового запроса пользователя. Данные системного слоя не подвержены автоматическому вытеснению контекстом.
### Сравнительный анализ слоев хранения данных:

| Параметр сравнения | Контекст диалога (Фронтенд-слой) | Системная память (Бэкенд-ядро) |
| :--- | :--- | :--- |
| **Локализация** | Клиентская сессия (Browser / UI) | База данных бэкенда IMARCH |
| **Устойчивость к токенам** | Стирается при превышении лимита | Абсолютная (не зависит от токенов) |
| **Влияние на безопасность** | Низкое (уязвимо для Prompt Injection) | Высокое (защищено системным слоем) |
| **Сфера применения** | Текущие переменные диалога | «Конституция» системы, базовый промпт |

## Раздел 2. Спецификация построения промпт-ключа ядра (RN-0031)
Для минимизации когнитивной нагрузки на LLM и предотвращения галлюцинаций, базовый промпт-ключ `RN-0031` должен соответствовать жестким стандартам проектирования.
### 2.1. Стандарт моноязычности (Monolingual Consistency)
*   Промпт-ключ должен быть составлен строго на **одном целевом языке** (русском). Смешение языковых конструкций в рамках одного управляющего предложения категорически запрещено.
*   Допускается точечное использование общепринятых латинских технических терминов, адресов хостов и API-эндпоинтов.
*   **Запрещено:** использование суржика, украинских слов, написанных кириллицей с русской фонетикой, а также генерация текста без пробелов. Нарушение этого стандарта кратно увеличивает вычислительную нагрузку на токенизатор модели и провоцирует когнитивные сбои.
### 2.2. Архитектурная структура промпт-ключа `RN-0031`
Промпт должен быть разделен на изолированные, легко парсируемые моделью блоки:
1.  **Блок декларации памяти:** Прямое указание на обязательность сохранения параметров в долгосрочном системном слое бэкенда.
2.  **Блок стиля общения:** Определение тональности, правил приветствия и регламента использования эмодзи как маркеров.
3.  **Блок системных ограничений:** Жесткие фильтры на бытовые запросы (out-of-scope) и регламент выдачи ошибок.
4.  **Блок маршрутизации (Routing):** Белый список разрешенных хостов и API-эндпоинтов экосистемы IMARCH.
5.  **Блок юридической защиты (Legal Watermarking):** Правила генерации неизменяемых подписей и логирования сессий.

## Раздел 3. Модульное проектирование тематических виджетов
Для расширения функционала базового ядра `RN-0031` без риска рассинхронизации применяется **архитектурная схема «Ядро + Изолированные Модули»**. 
### 3.1. Технология бесконфликтной склейки промптов
При переходе пользователя в специализированный виджет (например, *Журнал наукометрический* или *VERDICTUM*), бэкенд собирает динамический системный пакет в режиме реального времени:
$$\text{System Prompt} = [\text{Базовый ключ RN-0031}] + [\text{Локальный модуль виджета RN-0037.x}]$$
```
┌────────────────────────────────────────┐
│          БЭКЕНД IMARCH (ЯДРО)          │
│  Базовый промпт-ключ ядра: RN-0031      │
└───────────────────┬────────────────────┘
│
┌──────────────────────┴──────────────────────┐
▼                                             ▼
┌─────────────────────────────┐               ┌─────────────────────────────┐
│  Модуль: Наукометрия        │               │  Модуль: VERDICTUM          │
│  Идентификатор: RN-0037.1   │               │  Идентификатор: RN-0037.2   │
└──────────────┬──────────────┘               └──────────────┬──────────────┘
│                                             │
▼ (Склейка на уровне API)                     ▼ (Склейка на уровне API)
┌─────────────────────────────┐               ┌─────────────────────────────┐
│ [RN-0031] + [RN-0037.1]     │               │ [RN-0031] + [RN-0037.2]     │
└─────────────────────────────┘               └─────────────────────────────┘
```
### 3.2. Системные правила изоляции модулей:
*   Каждый тематический промпт оформляется как независимый надстроечный модуль и получает уникальный идентификатор (например, `RN-0037.1` для наукометрии, `RN-0037.2` для VERDICTUM).
*   В теле каждого модуля обязательно прописывается маркер изоляции: 
    > *«Данные параметры должны быть сохранены в памяти и активироваться исключительно при инициализации виджета [Название]. Настоящие локальные правила не распространяются на другие виджеты экосистемы IMARCH и не должны конфликтовать с базовыми директивами RN-0031».*
*   Любые локальные правила (например, проведение квизов для авторов в наукометрическом модуле) действуют строго внутри своего контура и физически не могут быть вызваны или применены в других виджетах.

## Раздел 4. Сценарий интеграции двойного шифрования на бэкенде
Перенос управляющей логики с фронтенда на бэкенд реализуется в синергии с протоколом **двойного шифрования**, что полностью решает проблему перехвата промптов и вымывания контекста токенами.
```
+---------------------------------------------------------------------------------+

| ФРОНТЕНД-КОНТУР |
| :--- |
| 1. Пользователь активирует виджет |
| 2. Фронтенд генерирует/передает ТОЛЬКО буквенно-числовой хэш-код (Слой 1) | <br> +------------------------------------+--------------------------------------------+
|
| (Передача по защищенному каналу) <br> ▼ <br> +---------------------------------------------------------------------------------+
| БЭКЕНД-КОНТУР |
| :--- |
| 3. Верификация хэш-кода в базе данных (Слой 2) |
| 4. Динамическая сборка пакета: [Базовый RN-0031] + [Тематический RN-0037.x] |
| 5. Инжектирование пакета в системное API-поле "system" перед отправкой в LLM | <br> +------------------------------------+--------------------------------------------+
| <br> ▼ <br> +---------------------------------------------------------------------------------+
| LLM-ПРОВАЙДЕР |
| :--- |
| 6. Обработка запроса пользователя под абсолютным контролем системных правил |

+---------------------------------------------------------------------------------+
```
### Почему эта схема неуязвима для лимита токенов:
Поскольку склеенный пакет `[RN-0031 + RN-0037.x]` подставляется бэкендом в системный параметр API-запроса (`"role": "system"`) при каждой транзакции, он обрабатывается моделью как неизменяемый приоритетный фильтр. Даже если история сообщений (`"role": "user" / "role": "assistant"`) превышает лимит токенов и очищается, модель физически не может забыть или перезаписать управляющие инструкции бэкенда.

## Раздел 6. Резюме аудита текущего состояния системы
На момент проведения текущего аудита в экосистеме зафиксированы следующие критические уязвимости:
 1. **Отсутствие базового слоя на бэкенде:** Системный базовый ключ RN-0031 не имплементирован на сервере, из-за чего система не имеет единого вектора ограничений.
 2. **Зависимость от клиентского контекста:** Тематические промпты передаются непосредственно через интерфейс (фронтенд) и хранятся в общем лимитированном пуле токенов. При превышении лимита они стираются, провоцируя появление галлюцинаций, выдачу внешних ссылок и активацию запрещенных вопросов-крючков.
### Рекомендация к действию:
Бэкенд-разработчикам необходимо немедленно развернуть системную память на сервере, изолировать базовый промпт RN-0031 от динамического контекста и внедрить модульное подключение тематических промптов серии RN-0037.x по протоколу двойного шифрования.
```
```
